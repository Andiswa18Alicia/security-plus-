General Security Concepts
Cryptography, PKI, authentication, security controls, and foundational principles.
Q 1.1MATCH
Match each cryptographic algorithm to its correct category.
Algorithm
AES-256
RSA-2048
SHA-256
RC4
Category Definitions
Symmetric Encryption — Same key to encrypt and decrypt
Asymmetric Encryption — Uses a public/private key pair
Hashing — One-way fixed-length output; integrity only
Stream Cipher — Encrypts one bit/byte at a time; deprecated
AES-256 — symmetric block cipher, same key encrypts/decrypts. RSA-2048 — asymmetric, public key encrypts, private key decrypts. SHA-256 — hashing, one-way 256-bit digest, no decryption. RC4 — stream cipher, deprecated (WEP, SSL), broken due to statistical bias.
Q 1.2ORDER
Arrange the PKI hierarchy from highest trust (top) to lowest (bottom).
  • ⠿Intermediate CA Certificate
  • ⠿Root CA Certificate
  • ⠿End-Entity (Leaf) Certificate
  • ⠿Trust Anchor (Root Store)
Trust Anchor (browser/OS root store) → Root CA (self-signed, offline) → Intermediate CA (online, signs leaf certs) → End-Entity Certificate (actual TLS/user cert). A broken chain at any level = untrusted certificate.

Q 1.4SELECT
Scenario: A financial organisation wants to eliminate "trusted network zones" and verify every request regardless of origin.
Which security model BEST describes this approach?
  • ADefense in Depth — layering multiple security controls
  • BZero Trust Architecture — "never trust, always verify"
  • CNetwork Segmentation — dividing the network into zones

Q 1.5CLI
Scenario: You are a sysadmin setting up a test web server. You need to generate a self-signed TLS certificate valid for 365 days using OpenSSL, outputting the key to server.key and cert to server.crt.
Type the correct OpenSSL command below:
$
openssl req -x509 -newkey rsa:2048 -keyout server.key -out server.crt -days 365 -nodes

req: certificate request. -x509: output self-signed cert instead of CSR. -newkey rsa:2048: generate new 2048-bit RSA key. -keyout server.key: write private key. -out server.crt: write certificate. -days 365: valid for 1 year. -nodes: no passphrase on key (needed for automated server startup).

Scenario: An attacker sends a carefully crafted email to the CFO of a company, referencing her name, her assistant's name, and a real upcoming board meeting. The email asks her to urgently wire $500,000 to a new vendor account.
This attack is BEST described as:
  • APhishing — bulk untargeted email attack
  • BWhaling — targeted spear phishing against a senior executive
  • CVishing — voice-based social engineering
  • DSmishing — SMS-based phishing attack
Whaling = spear phishing targeting C-level executives (CEO, CFO, CTO). The personalised details (name, assistant, real meeting) indicate OSINT research — classic whaling setup. Q 2.4MATCH
Match each attacker action to the correct MITRE ATT&CK tactic.
Attacker Action
Attacker sends phishing email to gain credentials
Attacker creates a new admin account to survive reboot
Attacker exploits local privilege to become SYSTEM
Attacker uses stolen creds to move to finance server
Attacker uploads data to external cloud storage
MITRE ATT&CK Tactics
Initial Access — first entry into the environment
Persistence — maintain foothold across reboots
Privilege Escalation — gain higher-level permissions
Lateral Movement — move between systems internally
Exfiltration — steal data out of the environment
MITRE ATT&CK framework describes adversary behaviour in 14 tactics. Knowing these tactics is essential for Security+ exam and real SOC work.

Q 3.2SELECT
Scenario: A company wants to move its email to the cloud. They want the provider to manage everything — servers, storage, OS, and the email application itself. The company only manages user accounts and data.
Which cloud service model BEST fits this requirement?
  • AIaaS — Infrastructure as a Service
  • BPaaS — Platform as a Service
  • CSaaS — Software as a Service
  • DFaaS — Function as a Service
SaaS: provider manages everything including the application. Customer only manages data and users. Examples: Microsoft 365, Gmail, Salesforce. PaaS: provider manages OS + runtime, customer manages application code (e.g. Heroku). IaaS: provider manages hardware + virtualisation, customer manages OS upwards (e.g. AWS EC2). Memory aid: SaaS = Somebody else's app. PaaS = Platform ready, you code. IaaS = Infrastructure only.
Q 3.3ORDER
Scenario: A packet arrives at a stateful firewall. Arrange the order in which the firewall evaluates the packet.
Drag to arrange from first checked (top) to last (bottom).
  • Match against ACL rules (top-down)
  • Check connection state table (existing session?)
  • Apply implicit deny (no rule matched)
  • Validate packet headers (IP, port, protocol)
Stateful firewall evaluation: 1-State table (if packet belongs to established session, allow immediately — most efficient). 2-Header validation (is the packet well-formed? Right flags?). 3-ACL rules (match top-down, first match wins). 4-Implicit deny (all firewalls end with "deny all" — if nothing matched, drop the packet). This is why rule order matters — a broad allow rule early can bypass later deny rules.

Q 4.2CLI
Scenario: You are a security analyst performing a vulnerability scan. You need to run an nmap SYN scan (stealth scan) against 192.168.1.0/24, scanning only the top 100 ports, with OS detection enabled, and output to a file called scan.txt.
Type the correct nmap command:
$
nmap -sS -O --top-ports 100 192.168.1.0/24 -oN scan.txt

-sS: SYN scan (stealth/half-open, doesn't complete TCP handshake, harder to detect in logs). -O: OS detection. --top-ports 100: scan the 100 most common ports. 192.168.1.0/24: target subnet (CIDR). -oN scan.txt: normal output to file. Other output formats: -oX (XML), -oG (grepable), -oA (all formats).

Q 5.2ORDER
Arrange the steps of a formal quantitative risk assessment in the correct order.
  • Calculate Annual Loss Expectancy (ALE = SLE × ARO)
  • Identify and value assets
  • Implement controls if ALE exceeds control cost
  • Calculate Single Loss Expectancy (SLE = Asset Value × EF)
  • Identify threats and calculate Exposure Factor (EF)
Quantitative formula chain: 1-Asset Value → 2-Exposure Factor (what % lost per incident?) → 3-SLE = AV × EF (loss per single event) → 4-ALE = SLE × ARO (ARO = Annual Rate of Occurrence; expected annual loss) → 5-Control decision (only implement if annual control cost < ALE reduction). Example: $1M server, 40% EF, 2×/year → SLE=$400K, ALE=$800K.
